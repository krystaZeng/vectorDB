# 完善代码 0526

## 一、整理范围

本文件整理 2026-05-26 前后新增和完善的代码思路，主要覆盖三类问题：

1. `payload-only update` 可能制造无意义 `DEAD outbox`。
2. Qdrant 已存在 collection 时缺少配置一致性校验。
3. 手工注册 vector column 可能直接写入脏元数据并标记为可用。

整体目标不是单点修补，而是把“进入可用状态”的条件收紧，让数据面、控制面、Qdrant 实际状态保持一致。

## 二、总体设计原则

### 1. ACTIVE 只代表正确性可用

P0 阶段不把 index ready、HNSW、quantization、on_disk 等性能条件放进 `ACTIVE` 判断。

当前语义：

- `ACTIVE`：可以正确写入、同步、查询，不会因为基础配置错误导致失败。
- `READY`：collection 元数据和 Qdrant 实际配置一致。
- index ready / search ready：后续 P1 再单独建模。

这样可以避免 `ACTIVE` 语义过重，也避免因为索引构建状态影响基础写入能力。

### 2. 所有 READY/ACTIVE 前置校验必须可复用

Qdrant collection 的 readiness 校验不应该只存在于 create table 链路。

目前抽象为统一能力：

- `VectorCollectionReadinessVerifier`
- `VectorEngineAdminPort.verifyCollection`
- `VectorEngineAdminPort.verifyAlias`

只要 metadata 状态准备进入可用态，就应该复用这套校验，避免不同入口绕过规则。

### 3. 手工注册不直接进入 ACTIVE

手工注册本质是声明元数据，不代表真实表结构、真实 Qdrant collection、alias 都已经正确。

因此 REST 手工注册默认进入 `BUILDING`，再通过 `verify-and-activate` 显式激活。

## 三、payload-only update 修复思路

### 问题

之前更新普通 payload 字段时，只要命中同步字段就会 enqueue `UPSERT`。

如果该行最初是 scalar-only insert，没有 vector，worker 重新读取行后会发现 vector 为空，然后把 outbox 标成 `DEAD`。

这会导致：

- 普通字段更新被误认为向量同步失败。
- 失败率指标被污染。
- 用户误判 update 链路不稳定。

### 当前实现

update 链路现在先读取行状态：

- 行是否存在。
- 当前行是否已有 vector。
- 本次请求是否包含 vector。
- 本次请求是否只更新 payload。
- 本次 payload 是否命中 Qdrant 同步字段。

之后再判断是否需要 Qdrant sync：

- 有 vector 写入：需要同步。
- 没有 vector，但当前行已有 vector，并且 payload 命中同步字段：需要同步 payload。
- 没有 vector，当前行也没有 vector：只更新关系表，不 enqueue outbox。

新增/使用的状态语义：

- `VECTOR_SYNC_ENQUEUED`
- `RELATIONAL_UPDATED_VECTOR_SYNC_SKIPPED_NO_VECTOR`
- `RELATIONAL_UPDATED_VECTOR_SYNC_NOT_REQUIRED`

### null vector 语义

当前 P0 明确不支持通过 `vector=null` 删除 Qdrant point。

REST update 如果显式传入：

```json
{
  "vector": null
}
```

直接返回 `INVALID_VECTOR_VALUE`。

如果后续要支持 `vector=null`，必须明确设计为：

- 关系表 `vector_col` 置 `NULL`
- enqueue `DELETE`
- Qdrant 删除 point 或删除指定 named vector

不能让 `null` 在“忽略字段”和“删除向量”之间语义模糊。

### 历史脏 outbox 兜底

worker 仍然保留防御逻辑：

- 如果手工或历史数据中存在 `UPSERT outbox`
- 但当前行 vector 为空

worker 会将事件标记为 `DEAD`，错误码为 `VECTOR_MISSING`。

这不是正常业务路径，而是用于识别历史脏数据或异常写入。

## 四、Qdrant collection readiness 校验

### 问题

之前 `ensureCollection` 只判断 collection 是否存在。

如果 Qdrant 已有同名 collection，但维度、distance、vector name 不一致，metadata 仍可能被标记为 `READY`。

后续写入或查询才失败，问题暴露太晚。

### 当前实现

`VectorEngineAdminPort` 增加：

- `verifyCollection`
- `verifyAlias`

`QdrantVectorEngineAdminAdapter` 在 collection 已存在时会读取 Qdrant 实际配置并校验：

- vector mode：unnamed / named
- vector name
- dimension
- distance

并且在 create collection 遇到 409 冲突后，不直接视为成功，而是再次 GET collection 配置并执行同一套校验。

### unnamed vector 规则

当前系统内部约定：

- `qdrantVectorName == null`
- `qdrantVectorName == blank`
- `qdrantVectorName == "default"`

以上都表示 Qdrant unnamed vector。

这意味着当前路径不支持 Qdrant named vector 名字真的叫 `"default"`。

这个规则已经显式收敛在代码方法里，避免散落在不同分支中形成隐式约定。

长期更稳妥的设计是增加 metadata 字段：

- `vector_mode = UNNAMED / NAMED`
- `qdrant_vector_name = null / actual_name`

这样就不需要用 `"default"` 间接表达 unnamed vector。

### named vector 匹配规则

vector name 按精确字符串匹配，不做大小写归一。

也就是说：

- `"image"` 和 `"IMAGE"` 不是同一个 named vector。

distance 可以归一：

- `Cosine`
- `COSINE`
- `cosine`

统一视为 `COSINE`。

### extra named vectors

P0 当前允许 Qdrant collection 中存在额外 named vectors。

例如系统期望：

```json
{
  "image": {
    "size": 512,
    "distance": "Cosine"
  }
}
```

实际 Qdrant 中存在：

```json
{
  "image": {
    "size": 512,
    "distance": "Cosine"
  },
  "text": {
    "size": 768,
    "distance": "Cosine"
  }
}
```

当前视为通过，只管理 expected vector。

如果产品后续要求一个 vector column 独占一个 collection，可以在 P1 增加 `strictCollectionMode` 配置。

### alias 校验

alias 校验和写入目标绑定。

当前规则：

- 如果 collection metadata 配置了 `aliasName`，则必须校验 alias 存在并指向当前 collection。
- 如果没有使用 alias，而是直接写 collection name，则不强制要求 alias。

## 五、手工注册 vector column 的长期 P0 方案

### 问题

之前手工注册入口只做简单 trim / uppercase，且可以直接注册成 `ACTIVE`。

这会造成：

- metadata 显示可用，但真实表不存在。
- 主键列或 vector 列不存在。
- Qdrant collection 未准备好。
- 数据 API 后续才失败。

### 当前实现

#### 1. 注册默认 BUILDING

REST 手工注册默认状态为 `BUILDING`。

手工注册请求如果显式传入：

- `ACTIVE`
- `FAILED`

会被拒绝。

内部链路仍保留兼容能力，避免影响 create table 这种“先写 metadata、后建真实表”的流程。

#### 2. 补 SQL identifier 校验

注册时校验：

- schema name
- table name
- primary key column
- vector column

避免把非法标识符写进 metadata。

#### 3. 补真实表结构校验

手工注册路径会校验：

- table 是否存在。
- primary key column 是否存在。
- vector column 是否存在。

只有关系表结构存在，才允许写入 `BUILDING` metadata。

#### 4. 新增 verify-and-activate 接口

新增接口：

```http
POST /api/v1/vector-columns/{columnId}/verify-and-activate
```

激活流程：

1. 读取 vector column metadata。
2. 如果已是 `ACTIVE`，幂等返回 already active。
3. 如果是 `DISABLED`，拒绝激活。
4. 只允许 `BUILDING` / `FAILED` 继续验证。
5. 校验真实表、主键列、vector 列。
6. 查找对应 `ACTIVE + READY` collection。
7. 复用 `VectorCollectionReadinessVerifier` 校验 Qdrant collection / alias。
8. 成功后条件更新为 `ACTIVE`。
9. 失败后保持或更新为 `FAILED`，并记录错误信息。

#### 5. 条件状态更新

激活最后一步不是无条件：

```sql
UPDATE SYS_VECTOR_COLUMNS_
SET STATUS = 'ACTIVE'
WHERE COLUMN_ID = ?
```

而是条件更新：

```sql
UPDATE SYS_VECTOR_COLUMNS_
SET STATUS = ?,
    REMARK = ?,
    UPDATED_AT = CURRENT_TIMESTAMP
WHERE COLUMN_ID = ?
  AND STATUS IN (?, ?)
```

当前允许的前置状态是：

- `BUILDING`
- `FAILED`

这样可以避免激活流程和 disable / repair / retry 并发时覆盖对方状态。

### ACTIVE 幂等策略

P0 选择简单幂等：

- 当前已经 `ACTIVE`：直接返回 already active。
- 不在重复点击 activate 时做 drift detection。

原因是避免用户重复操作导致正在服务的 column 被意外标记为 `FAILED`。

后续 P1 可以新增单独接口：

```http
POST /api/v1/vector-columns/{columnId}/reverify
```

专门用于漂移检测。

## 七、测试覆盖

新增和补强的测试主要覆盖：

### update / outbox

- payload-only update 且当前行无 vector，不 enqueue outbox。
- payload-only update 且当前行已有 vector，正常 enqueue outbox。
- 显式 `vector=null` 被拒绝。
- 历史脏 `UPSERT outbox + NULL vector` 被标记 `DEAD`，错误码 `VECTOR_MISSING`。

### Qdrant readiness

- 已存在 unnamed collection，dimension / distance 一致时通过。
- dimension 不一致失败。
- distance 大小写归一。
- named vector 精确匹配。
- named vector 缺失失败。
- expected vector 存在且有 extra named vectors 时通过。
- named `"default"` 和 unnamed 语义冲突时失败。
- 409 后执行 GET + validate。

### 手工注册与激活

- 手工注册默认 `BUILDING`。
- 非法 identifier 被拒绝。
- 真实表不存在被拒绝。
- 手工直接注册 `ACTIVE` 被拒绝。
- `verify-and-activate` 成功后进入 `ACTIVE`。
- 缺少 ready collection 时激活失败，并将 column 标记为 `FAILED`。

## 八、当前方案评价

从产品化角度看，这次修改把方案从“可用但容易产生脏状态”推进到了“P0 可控、可解释、可排查”。

当前更稳妥的地方在于：

- 数据面不会因为普通字段更新制造无意义同步失败。
- Qdrant collection 不会只凭名字存在就被误判为 ready。
- 手工注册不再绕过真实表和 Qdrant 校验。
- 状态流转从直接写 `ACTIVE` 改成 `BUILDING -> verify -> ACTIVE`。
- 并发激活通过条件更新减少状态覆盖风险。

仍未完全产品化的地方：

- 缺少 `verified_at`。
- 缺少结构化 `readiness_error_code` / `readiness_error_message`。
- 缺少主动 `reverify` 接口。
- 缺少 drift detection。
- 缺少 `SEARCH_READY` / `INDEX_READY` 等性能状态。
- `"default"` 表达 unnamed vector 仍是过渡方案。

## 九、下一步建议

P1 建议按下面顺序推进：

1. 给 vector column / collection metadata 增加 `verified_at`、`readiness_error_code`、`readiness_error_message`。
2. 新增 `reverify` 接口，专门检测 ACTIVE 资源是否发生配置漂移。
3. 增加 drift detection 定时任务，定期校验 Qdrant 实际配置和 metadata 是否一致。
4. 增加 `vector_mode` 字段，替代 `"default"` 表示 unnamed vector 的过渡约定。
5. 增加 `SEARCH_READY` / `INDEX_READY`，把性能可查询状态和正确性可写状态拆开。
6. 按产品语义决定是否增加 `strictCollectionMode`。
